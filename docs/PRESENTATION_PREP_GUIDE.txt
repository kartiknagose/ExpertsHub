EXPERTSHUB V2 - PRESENTATION PREP GUIDE (EASY LANGUAGE)
Date: April 8, 2026

============================================================
1) PROJECT IN ONE LINE
============================================================
ExpertsHub is a home-service platform where customers book workers
(like plumber, electrician, cleaner), workers complete jobs, and admin
manages quality, safety, payments, and growth.


============================================================
2) WHAT PROBLEM THIS PROJECT SOLVES
============================================================
Before platforms like this, users face these issues:
- Hard to find trusted workers quickly.
- No clear pricing.
- No proper safety in emergencies.
- Manual payment and no proper job tracking.

ExpertsHub solves this by giving:
- Verified workers.
- Smart booking with OTP verification.
- Live updates and chat.
- Secure payment flow.
- Admin control with analytics and audit.


============================================================
3) HIGH-LEVEL SYSTEM ARCHITECTURE
============================================================
This project has 3 main layers:

1. Frontend (client)
   - React web app used by customer, worker, and admin.

2. Backend (server)
   - Node.js + Express API.
   - Handles business logic, auth, booking, payments, etc.

3. Data and infra
   - PostgreSQL (main database via Prisma ORM).
   - Redis (cache + real-time scaling support).

Also integrated:
- Socket.IO for real-time features.
- Razorpay for payment/payout flow.
- Cloudinary for media uploads.
- Sentry for error monitoring.


============================================================
4) FRONTEND TECHNOLOGIES (WHAT AND WHY)
============================================================
Core stack:
- React 18: main UI framework.
- Vite: fast development and production build.
- React Router: role-based pages and protected routes.
- Tailwind CSS: utility-based styling.
- React Query: API caching, retry, and data fetching state.
- Axios: API communication.
- Zustand: lightweight client state where needed.
- React Hook Form + Zod: forms + validation.
- i18next: multilingual support.
- Leaflet / React-Leaflet: maps and live location UI.
- Socket.IO client: real-time events.
- Sonner: toast notifications.
- Framer Motion: smooth UI animations.
- Recharts: dashboards and charts.

PWA support:
- vite-plugin-pwa, service worker, offline page, install prompt,
  update prompt, push support.

Frontend structure concept:
- pages/: route-level screens.
- components/: reusable UI and feature components.
- api/: all API service calls.
- context/: auth, theme, notification, city, SOS contexts.
- hooks/: reusable logic.


============================================================
5) BACKEND TECHNOLOGIES (WHAT AND WHY)
============================================================
Core stack:
- Node.js + Express 5: API server.
- Prisma ORM + PostgreSQL: schema-first relational data handling.
- Redis + ioredis: caching/session support and scaling utilities.
- Socket.IO: real-time communication.
- JWT + cookies + bcryptjs: secure auth.
- Helmet, CORS, rate-limit, validation middleware: API security.
- Winston + Morgan: structured logging.
- Prometheus client: metrics endpoint for observability.
- Node-cron: scheduled jobs (like payout automation).
- Multer + Sharp + Cloudinary: upload pipeline.
- Razorpay SDK: orders, payment verification, refunds, payout flow.
- Nodemailer + fallbacks: email flows.
- Web Push (VAPID): browser push notifications.
- Sentry Node: production error monitoring.

Backend design style:
- Module-based architecture.
- Each module has routes, controller, service, schema/validation.

Main backend modules include:
- auth, workers, services, bookings, customers
- availability, verification, admin
- payments, payouts, invoices
- safety, location, notifications, chat
- analytics, growth, ai, uploads


============================================================
6) DATABASE DESIGN (IMPORTANT FOR PRESENTATION)
============================================================
Database: PostgreSQL
ORM: Prisma

Key entities and why they matter:
- User: customer/worker/admin identity and profile state.
- WorkerProfile: worker-specific data (skills, rates, radius, payout fields).
- Service: list of services and pricing metadata.
- Booking: heart of business flow (status, schedule, price, OTP, location).
- Payment: payment records linked to booking/customer.
- Review: quality feedback after job.
- Availability: worker weekly time slots.
- WorkerLocation: live worker coordinates.
- WorkerVerificationApplication + docs/media/reference: trust/KYC layer.
- Notification + PushSubscription + NotificationPreference: communication layer.
- SOSAlert + EmergencyContact: safety layer.
- Conversation + Message: chat layer.
- Invoice: tax/compliance and downloadable bill.
- Wallet/loyalty/referral related tables: growth and retention features.

Important enums used in flow:
- Role: CUSTOMER, WORKER, ADMIN
- BookingStatus: PENDING, CONFIRMED, IN_PROGRESS, COMPLETED, CANCELLED
- PaymentStatus: PENDING, PAID, FAILED, REFUNDED
- VerificationStatus: PENDING, APPROVED, REJECTED, MORE_INFO

Why this schema is strong:
- Clear role separation.
- Complete booking lifecycle support.
- Financial traceability (booking + payment + payout + invoice).
- Safety and audit-friendly data.


============================================================
7) AUTHENTICATION AND AUTHORIZATION WORKFLOW
============================================================
Simple flow:
1. User registers and verifies email.
2. User logs in; backend issues secure auth cookie/JWT.
3. Protected routes check authentication.
4. Role middleware restricts access:
   - Customer pages/actions
   - Worker pages/actions
   - Admin operations

Security layers used:
- Password hashing with bcrypt.
- Helmet headers.
- CORS policy.
- Rate limiting on sensitive endpoints.
- Request validation before business logic.


============================================================
8) BOOKING WORKFLOW (END-TO-END)
============================================================
This is one of the most important sections to explain.

Step-by-step booking lifecycle:
1. Customer checks price preview (dynamic pricing engine).
2. Customer creates booking with service/time/location.
3. Booking starts as PENDING.
4. Worker accepts -> booking becomes CONFIRMED.
5. System generates start OTP.
6. Worker reaches site and enters start OTP -> IN_PROGRESS.
7. Worker uploads work proof photos (before/after).
8. Worker enters completion OTP -> COMPLETED.
9. Customer reviews and rates worker.

Extra booking capabilities:
- Cancel and reschedule support.
- Session management for recurring/multi-visit jobs.
- Booking status history tracking.
- Cancellation policy endpoint and penalty logic.

Why this workflow is strong:
- OTP prevents fake status updates.
- Session and history improve transparency.
- Multi-state lifecycle fits real service operations.


============================================================
9) PAYMENT, REFUND, AND PAYOUT WORKFLOW
============================================================
Customer payment flow:
1. Backend creates Razorpay order.
2. Frontend opens checkout.
3. Payment success is validated (webhook/signature flow).
4. Booking payment status updates to PAID.

Refund flow:
- Based on cancellation timing policy.
- Refund can be full or partial (as per rules).
- Payment status can move to REFUNDED.

Worker payout flow:
1. Platform calculates commission and worker share.
2. Worker earnings tracked in profile/wallet context.
3. Worker can request instant payout (when eligible).
4. Scheduled payout jobs also supported.
5. Payout history is available.

Financial trust points to mention:
- Clear split of total amount, commission, worker payout.
- Payment and payout are auditable records.
- Invoice generation for compliance.

19) QUICK Q&A PREP (LIKELY QUESTIONS)
============================================================
Q1) How do you prevent fake completion?
A) OTP verification at start and completion, plus photo proof and status logs.

Q2) How do you secure APIs?
A) Auth middleware, role checks, validation, rate limits, helmet, CORS,
   and structured error handling.

Q3) Why Redis?
A) Cache/session help and real-time scaling support for socket adapter.

Q4) How do payments stay reliable?
A) Server-side order creation, webhook verification, status tracking,
   refund support, and payout history.

Q5) Is it scalable?
A) Yes, modular backend, relational schema, Redis-based socket scaling,
   and containerized deployment support.
